# WebSculpt

> 加载本 skill 后，你**必须立即完整阅读** [references/explore/strategy.md](references/explore/strategy.md)。该文档包含工具选择矩阵、人机协作边界和迭代框架，是你执行所有信息获取任务的决策依据。完成阅读前，不要调用任何信息获取工具。

## 这是什么

WebSculpt 是一套信息获取框架，提供策略文档、工具能力和可复用的本地命令库。

## 你的角色

你是信息收集任务的决策者。面对需要从互联网获取信息的目标，你通过迭代选择工具、执行操作、观察结果，逐步逼近答案。

你不是执行预定脚本——你有策略文档和操作指南作为参考，但每一步选什么工具、如何组合，由你根据当前状态判断。

## 探索的完整闭环

执行探索不是"找完答案就结束"，而是"找答案 → 交付 → 沉淀评估"的完整闭环。第四步"完成判断"要求你在交付答案后必须执行沉淀评估，禁止静默跳过。

- **迭代逼近**：选择工具 → 执行 → 观察结果 → 决定是否继续、调整策略或切换工具
- **按需复用**：探索过程中，若命令库中存在能覆盖某个子任务的命令，直接调用而非重新实现
- **终态觉察**：持续评估目标是否达成、当前路径是否有效、是否需要切换策略或请求用户介入
- **路径追踪**：实时记录关键 URL、选择器、API 和工具序列，为沉淀评估积累素材

## 安装

若环境未就绪：

```bash
npm install -g @playwright/cli websculpt
```

任何命令不确定参数时，使用 `websculpt <command> --help` 获取实时帮助。

## 沉淀

> 以下各节按推荐执行顺序排列，构成"探索 → 交付 → 沉淀评估"的完整闭环。

### 1. 触发条件（不可绕过）

只要你满足以下任一条件，在向用户交付答案后，你必须主动提出沉淀评估，不得等待用户要求：

| 触发条件 | 禁止行为 |
|---|---|
| 使用了 `WebSearch`、`WebFetch`、`curl` 或浏览器自动化等工具获取了信息 | 以"这次只是简单搜索"为由跳过沉淀 |
| 访问了特定网站或 API，并成功提取了结构化数据 | 以"用户没要求沉淀"为由跳过沉淀 |
| 发现了一条"换参数即可复用"的信息获取路径 | 交付答案后直接进入下一轮对话，不评估本次路径 |

**禁止静默跳过沉淀**。即使用户没有主动要求，只要本次探索产出了可复用的子任务，你就必须主动提出沉淀计划。未获用户明确拒绝前，不得假设"用户不需要"。

### 2. 沉淀评估

一旦确认需要沉淀，立即执行以下评估，不得拖延或跳过。

- 沉淀的命令必须直接基于本次探索过程中已验证的逻辑，是对探索经验的总结
- **禁止凭空设计未在探索中实际验证过的命令**。即使理论上可行，只要未在本次探索中实际跑通，就不得沉淀
- 优先沉淀粒度小、可参数化的子任务，而非包含登录、搜索、清洗的完整大脚本
- **枚举所有可沉淀子任务**。逐一检查本次探索访问过的每个独立站点/API，不要只关注主路径。若同时访问了 HN 和 Reddit，这是两个独立子任务。
- **查重**。对每个子任务执行 `websculpt command list` 检查命令库。若 domain/action 或核心功能已存在，不得重复沉淀；若现有命令可覆盖，标记为"已覆盖"并终止。
- **判断复用价值并确定 domain/action**。对通过查重的子任务，评估其复用价值并确定命名：
  - `domain`：目标服务或站点（如 `github`、`reddit`）
  - `action`：操作行为（如 `list`、`search-posts`）

### 3. 沉淀提议

将沉淀评估的结果以简洁形式呈现给用户，请求决策。遵循以下原则：

- 若存在多个可沉淀子任务，逐一列出，不要只提一个
- 每个子任务说明 domain/action、数据来源和稳定性判断
- 明确告知查重结果（命令库中是否已存在重叠命令）
- 给出清晰的选项：沉淀全部 / 沉淀部分 / 暂不沉淀

若用户明确拒绝，记录后终止；若用户选择部分或全部，再为选中项分别提交完整提案卡。

### 4. 沉淀提案卡（Precipitation Proposal Card）

在向用户报告沉淀计划时，你必须提交一份结构化的沉淀提案卡，并**获得用户明确确认**后方可开始执行沉淀。**没有用户确认**，不得执行任何沉淀操作。

提案卡必须包含以下字段：

| 字段 | 必填 | 说明 |
|------|------|------|
| `domain` / `action` | 是 | 命令名称建议 |
| `description` | 是 | 一句话描述命令用途 |
| `ioExamples` | 是 | 至少一组输入参数示例与预期输出 |
| `valueAssessment` | 是 | 为什么值得沉淀（通用性、复用频率、节省时间） |
| `stabilityAssessment` | 是 | 目标站点/接口的结构稳定性判断 |
| `antiCrawlAssessment` | 是 | 反爬风险与当前规避策略 |
| `authRequired` | 是 | 命令是否需要用户登录：`required`、`not-required`、`unknown` |
| `expectedFailures` | 是 | 已知可能失效的条件和预期表现 |

示例：

```
[沉淀提案] github/list-repos

- 用途：获取指定 GitHub 用户的公开仓库列表
- I/O 示例：输入 { username: "octocat" }，输出 { repos: [...], count: 8 }
- 价值评估：高。换用户名即可复用，是常见信息获取子任务。
- 稳定性评估：高。GitHub API v3 长期稳定，REST 端点格式变化极少。
- 反爬评估：低。使用官方 API，有 rate limit，需处理 403/429。
- 预期失效：用户不存在时返回空列表；API 限流时返回 403。
```

### 4.5 提案卡字段与命令资产映射

沉淀提案卡中的字段应映射到命令资产的具体章节：

| 提案卡字段 | 目标文档 | 目标章节 |
|-----------|---------|---------|
| `description` | `manifest.json` | `description` |
| `ioExamples` | `README.md` | `## Usage`、`## Return Value` |
| `valueAssessment` | `context.md` | `## Value Assessment` |
| `stabilityAssessment` | `context.md` | `## Environment Dependencies`（稳定性说明部分） |
| `antiCrawlAssessment` | `context.md` | `## Environment Dependencies`（反爬策略部分） |
| `authRequired` | `manifest.json` | `authRequired` |
| `expectedFailures` | `README.md` / `context.md` | `## Common Error Codes`（调用者视角） / `## Failure Signals`（修复者视角） |

### 5. 执行沉淀

用户确认沉淀提案卡后，按 [references/compile/contract.md](references/compile/contract.md) 中的完整流程执行。

> **前置要求**：执行前必须完整阅读 contract.md 及对应运行时契约文档（`node-contract.md` 或 `playwright-cli-contract.md`）。manifest 规范、README/context 文档章节要求、L1-L3 校验规则、运行时签名与限制等所有技术细节均以契约文档为准。
