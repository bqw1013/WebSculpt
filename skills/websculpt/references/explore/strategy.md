# WebSculpt 探索策略

> 本文档面向 AI Agent，说明在面对陌生信息获取场景时，如何根据任务特征选择工具、制定探索策略，并为后续沉淀出稳定、可复用的命令资产。

---

## 1. 核心哲学：目标导向的探索

探索不是"按预定步骤执行"，而是"带着目标进入，边看边判断，遇到阻碍就解决，发现内容不够就深入"。全程围绕"我要达成什么"做决策。

### 四步决策框架

**① 定义成功标准**
先明确用户要做什么。什么算完成了？需要获取什么信息、执行什么操作、达到什么结果？这是后续所有判断的锚点。

**② 选择起点**
根据任务性质、达成条件，选一个最可能直达的工具作为第一步。一次成功最好；不成功则在③中调整。

**③ 过程校验**
每一步的结果都是证据，不只是成功或失败的二元信号。用结果对照①的成功标准，更新你对目标的判断：
- 路径在推进吗？
- 结果的整体面貌（质量、相关度、量级）是否指向目标可达？
- 发现方向错了调整策略，不在同一个方式上反复重试。

搜索没命中不等于"还没找对方法"，也可能是"目标不存在"；API 报错、页面缺少预期元素、重试无改善，都是在告诉你该重新评估方向。

遇到弹窗、登录墙等障碍，判断它是否真的挡住了目标：挡住了就处理，没挡住就绕过——内容可能已在页面 DOM 中，交互只是展示手段。

**④ 完成判断**
对照定义的成功标准，确认任务完成后才停止，但也不要过度操作，不为了"完整"而浪费代价。

## 2. 人机协作决策框架

探索过程中，你不需要独自解决所有问题。及时识别"这个障碍超出我当前能力或权限"并请求用户介入，是高效的策略选择，不是失败。

### 何时请求用户介入

以下情况倾向于停下来问用户：

- **需要用户凭据或授权**：访问目标需要登录、付费、订阅，而你没有活跃的会话或权限。
- **需要用户做出价值判断**：操作存在风险（如账号封禁可能），或需要在多个路径中做取舍，而你缺乏用户的偏好信息。
- **所有自动化路径都失效**：目标站点的结构或反爬策略已经改变，你现有的工具组合无法突破。

以下情况你应该自己解决，不问用户：

- **工具选择或切换**：这是你的决策范围，不需要用户批准。
- **单一方式的临时失败**：网络波动、页面加载慢等，重试或调整参数即可。
- **通过切换策略可以绕过的障碍**：如从静态抓取切换到浏览器自动化。

### 协作方式

请求用户时，说明三件事：
1. 当前进展（已经做了什么）
2. 遇到的障碍（什么问题导致无法继续）
3. 需要用户提供的帮助（具体做什么）

用户响应后，根据新信息继续探索，不重复之前的无效路径。

## 3. 工具选择策略

### 决策矩阵

| 场景 | 优先工具 | 说明 |
|------|---------|------|
| 需要发现信息来源、获取搜索摘要 | WebSearch | 信息定位入口 |
| URL 已知，需要从页面定向提取特定可读内容 | WebFetch / curl | 静态层直接获取 |
| URL 已知，需要原始 HTML 源码（meta、JSON-LD 等结构化字段） | curl | 获取未经处理的原始响应 |
| 静态方式已知无效的反爬严格站点 | 浏览器自动化 | 跳过静态层，直接启用浏览器 |
| 需要登录态、多步骤交互、或像人一样自由导航探索 | 浏览器自动化 | 复用用户浏览器会话 |

### 启发式规则

**先轻后重**
优先尝试 WebSearch、WebFetch 或 curl。只有在确认静态方式不可达（遇反爬、需交互、需登录态）时，才启用浏览器自动化。

**一手信息优先**
搜索引擎和聚合平台是信息**定位**工具，不是**证明**工具。找到来源后，直接访问原始页面读取原文。多个二手来源引用同一个错误会造成循环印证假象。

## 4. 复用已有命令

WebSculpt 维护一个本地命令库，其中包含已沉淀的信息获取路径。这些命令通常只覆盖特定子任务，而非完整目标。

### 何时检查命令库

在开始探索或分解任务后，判断当前子任务是否可能被已有命令覆盖：

```bash
websculpt command list
```

该命令返回所有可用的扩展命令，标注来源（builtin / user），包含 domain、action 和 description。

### 如何判断是否匹配

对比命令的 domain/action/description 与你的当前子任务。若某命令的功能描述直接对应你要完成的某一步，即可复用。

### 执行命令

```bash
websculpt <domain> <action> --<param> <value>
```

参数和用法可通过 `--help` 查看：

```bash
websculpt <domain> <action> --help
```

## 5. 探索与沉淀的衔接

探索的终点不是"跑通了"，而是"找到了一条在目标站点当前环境下可持续运行的路"。

### 为沉淀做准备的探索原则

探索阶段的路径选择直接影响后续能否沉淀为可复用命令。遵循以下原则：

- **优先寻找与前端实现解耦的路径**：数据接口优于 DOM 遍历，语义标识优于位置依赖。
- **规避临时状态**：不依赖仅在当前会话有效的标识或令牌。
- **规避硬编码时序**：不依赖固定等待时间或预设的操作顺序。
- **规避易变标识**：不依赖可能随站点更新而改变的类名或结构路径。

### 反爬友好度的考量

- **程序化探测速度快但痕迹明显**：高频 DOM 操作或快速翻页容易触发风控。探索阶段就要测试"什么节奏下站点会拒绝响应"，把这个节奏作为沉淀命令的默认参数。

- **用户行为模拟是反爬友好的基底**：交互操作有自动等待和真实事件流。但沉淀后如果批量调用，仍然可能被识别。探索时要验证单链路的可行性，也要估计批量调用的风险。

- **有些站点只有浏览器层才有内容**：静态方式完全无效，只能走浏览器自动化。探索时就要确认"必须用浏览器"这个事实，沉淀命令直接声明 `playwright-cli` runtime，不浪费代码在 fallback 上。
