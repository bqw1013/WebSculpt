# WebSculpt 探索策略

> 本文档面向 AI Agent，说明在面对陌生信息获取场景时，如何根据任务特征选择工具、制定探索策略，并为后续沉淀出稳定、可复用的命令资产。

---

## 1. 工具选择策略

### 第一步：检查命令库

WebSculpt 的命令库是本地已沉淀的信息获取路径集合。每条命令通常只覆盖一个特定子任务（如获取某个站点的热门列表、提取特定页面的结构化数据），不是完整工作流。在调用 WebSearch、WebFetch、curl 或浏览器自动化之前，先检查命令库中是否已有可用路径。

```bash
websculpt command list
```

返回所有可用命令，含 domain、action、description 和来源（builtin / user）。浏览列表寻找与你当前子任务相关的条目。

> 若需要结构化输出以便解析，在任何命令前追加 `--format json`，如 `websculpt --format json command list`。

若找到候选命令，按以下顺序由浅入深确认其适用性：

```bash
websculpt <domain> <action> --help
```

快速了解参数列表、默认值和基本用法。

```bash
websculpt command show <domain> <action>
```

查看完整契约卡片，包括参数详情、运行时前置条件、资产完整性。

```bash
websculpt command show <domain> <action> --include-readme
```

追加阅读 README.md 原文，获取用法示例、返回值说明和错误码定义。信息最完整，但体积最大，按需使用。

确认匹配后直接调用执行；若命令库无覆盖，再进入外部工具选择。

沉淀前同样须执行此检查，不得在未确认命令库无重叠的情况下提议沉淀新命令。

### 第二步：选择外部工具

当命令库无覆盖时，进入外部工具选择。

**先判断任务特征**

根据用户意图判断信息获取路径的复杂度，避免在明显需要浏览器环境的任务上浪费时间尝试轻方法：

- 若任务涉及登录态、多步骤交互、JS 渲染内容或需要像人一样自由导航，直接启用浏览器自动化。
- 若目标站点已知对静态抓取有严格反爬，同样直接启用浏览器自动化。
- 若任务只是查找公开信息或提取已知 URL 的静态内容，从轻方法开始。

场景参考：

| 场景 | 优先工具 |
|------|---------|
| 需要发现信息来源、获取搜索摘要 | WebSearch |
| URL 已知，需要页面定向提取可读内容 | WebFetch / curl |
| URL 已知，需要原始 HTML 源码（meta、JSON-LD 等） | curl |
| 需要登录态、多步骤交互、自由导航 | 浏览器自动化 |
| 已知反爬严格，静态方式已确认无效 | 浏览器自动化 |

**再确认约束**

当前浏览器自动化仅支持 `browser` 运行时。在使用前，你必须先阅读 `../access/playwright-cli/guide.md`，确认连接方式、可用命令与行为约束后再执行。

其他工具（WebSearch、WebFetch、curl 等）按你自身能力直接使用，无需查阅 access 层。

**执行中的原则**

- **特征不明确时从轻开始**：若任务未落入上述"直接启用浏览器"的情形，优先尝试 WebSearch、WebFetch 或 curl。但一旦收到结构性拒绝信号（CAPTCHA、登录墙、内容需交互才可见），立即切换，不在同一层级纠缠。
- **止损**：同一工具调整参数后无实质改善，切换至下一层级；切换工具不是失败。
- **一手信息优先**：搜索引擎只用于定位来源，找到后直接访问原始页面读取原文。

## 2. 人机协作决策框架

探索过程中，你不需要独自解决所有问题。及时识别"这个障碍超出我当前能力或权限"并请求用户介入，是高效的策略选择，不是失败。

### 何时请求用户介入

以下情况倾向于停下来问用户：

- **需要用户凭据或授权**：若任务涉及登录、付费或订阅，而你没有活跃的会话或权限，在尝试任何工具之前先问用户，避免在无权限的状态下浪费尝试。
- **需要用户在多个路径中做取舍**：当存在多个可行方案（如"用官方 API 有 rate limit 但稳定，用页面抓取无限制但易失效"），而你缺乏用户的偏好信息时。
- **所有合理工具组合均已失效**：当你已经尝试了命令库复用、轻量工具和浏览器自动化的完整组合，仍然无法突破目标站点的反爬或结构变更。

以下情况你应该自己解决，不问用户：

- **工具选择或切换**：这是你的决策范围，不需要用户批准。
- **单一方式的临时失败**：网络波动、页面加载慢等，重试或调整参数即可。
- **通过切换策略可以绕过的障碍**：如从静态抓取切换到浏览器自动化。

### 协作方式

请求用户时，说明三件事：

1. **当前进展**：已经做了什么
2. **遇到的障碍**：什么问题导致无法继续
3. **建议方案**：给出具体选项供用户选择，而非抛出开放性问题

用户响应后，根据新信息继续探索，不重复之前的无效路径。

## 3. 探索与沉淀的衔接

探索的终点不是"跑通了"，而是"交付答案 + 完成沉淀评估"。沉淀评估不可跳过。

探索过程中注意收集关键信息（工具序列、URL、选择器、反爬策略等），这些素材将用于填写 `context.md`。沉淀命令的编写规范（manifest 格式、README.md / context.md 章节要求、运行时契约）见 `../compile/contract.md`。

沉淀的执行流程（提案卡、draft、validate、create）见 SKILL.md。
