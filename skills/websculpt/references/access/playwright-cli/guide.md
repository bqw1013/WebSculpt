# Playwright CLI 接入模块

## 1. 概览

Playwright CLI（`@playwright/cli`）是一个基于命令行的浏览器自动化工具，封装了 Playwright 的高级 API（自动等待、智能定位器、会话管理等），通过简洁的 CLI 命令完成页面导航、元素交互、代码执行等操作。

### 核心机制

- **Snapshot**：每个命令执行后，CLI 返回当前页面的结构化快照，包含可交互元素的临时引用标识（ref，如 `e1`、`e2`）。
- **双模式操作**：既可通过原生 CLI 命令（`click`、`fill` 等）快速交互，也可通过 `run-code` 执行任意 Playwright 代码处理复杂逻辑。
- **灵活定位**：支持 snapshot ref、CSS 选择器、Playwright locator 三种元素定位方式。

### 前置检查

在尝试连接浏览器之前，按以下顺序确认环境就绪：

1. **确认 `playwright-cli` 可用**
   ```bash
   playwright-cli --version
   ```
   若未安装，引导用户执行以下命令全局安装：
   ```bash
   npm install -g @playwright/cli
   ```

2. **确认浏览器远程调试已开启**
   引导用户在 Chrome 或 Edge 中新建标签页，访问 `chrome://inspect/#remote-debugging`，勾选 **"允许此浏览器实例进行远程调试"**，并保持浏览器处于打开状态。

3. **展示风险须知**

   开始操作前，向用户直接展示以下提示：

   > 温馨提示：部分站点对浏览器自动化操作检测严格，存在账号封禁风险。已内置防护措施但无法完全避免，继续操作即视为接受。

### 常用命令速查

以下命令覆盖绝大多数自动化场景：

| 类别 | 命令 | 说明 |
|------|------|------|
| 导航 | `goto <url>` | 在当前页面导航到指定地址 |
| 感知 | `snapshot` | 获取页面结构化快照 |
| 感知 | `eval <expression>` | 在页面上下文中执行 JavaScript 表达式并返回结果 |
| 交互 | `click <target>` | 点击指定元素 |
| 交互 | `fill <target> <text>` | 在输入框填入文本 |
| 交互 | `type <text>` | 在聚焦元素上输入文本 |
| 交互 | `press <key>` | 模拟键盘按键（如 `Enter`、`ArrowDown`） |
| 高级 | `run-code "<code>"` | 执行 Playwright 代码片段，接收 `page` 对象 |
| 输出 | `screenshot` | 截取当前页面 |

完整命令列表和参数详情，请通过 `playwright-cli --help` 或 `playwright-cli <command> --help` 查阅。

### 元素定位方式

Playwright CLI 支持三种元素定位手段：

1. **Snapshot ref**（默认方式）
   执行 `snapshot` 后，页面中的可交互元素会被分配临时引用标识，后续命令可直接使用该标识操作元素。

2. **CSS 选择器**
   支持通过标准 CSS 选择器定位元素。

3. **Playwright locator**
   支持通过 Playwright 的 locator API 定位元素，例如 `getByRole`、`getByTestId` 等。

> **注意**：`run-code` 中应使用稳定选择器（CSS 选择器或 Playwright locator），不要使用临时 ref。Snapshot ref 仅在单次会话中有效，下次执行会重新分配。

## 2. CDP 连接

Playwright CLI **只能通过 CDP attach 到用户已手动开启远程调试的现有浏览器实例**。这是 WebSculpt 中唯一被允许的浏览器连接方式。AI 不得使用 launch、connect 或任何等效机制创建新浏览器实例或连接到其他浏览器进程。

### 连接步骤

1. **用户开启远程调试**
   在目标浏览器中新建标签页，访问 `chrome://inspect/#remote-debugging`，勾选 **"允许此浏览器实例进行远程调试"**，并保持浏览器处于打开状态。

   > Chrome 和 Edge 使用同一设置入口和流程。

2. **执行 attach**

   ```bash
   # 附加到 Chrome
   playwright-cli attach --cdp=chrome
   
   # 附加到 Microsoft Edge
   playwright-cli attach --cdp=msedge
   ```

   连接成功后，CLI 返回当前页面 snapshot。

### 连接失败排查

如果 `attach` 失败，按以下顺序排查：

1. **确认远程调试已开启**
   请用户确认浏览器仍在运行，且已访问 `chrome://inspect/#remote-debugging` 并勾选远程调试。若未开启，返回上一步重新设置。

2. **排查后台进程冲突**
   Chrome/Edge 为多进程架构，关闭前台窗口后后台实例仍可能占用调试端口。请用户确认已关闭所有浏览器窗口（包括系统托盘图标）。

   若确认已完全关闭但仍无法 attach，则向用户说明情况并请求授权终止所有浏览器进程。获得明确授权后：
   - 强制终止所有 Chrome/Edge 进程
   - 引导用户重新启动浏览器并完成第 1 步的远程调试设置
   - 再次执行 `attach`

   > 强制终止浏览器进程会丢失未保存的标签页和数据，必须获得用户明确授权后方可执行。

### 登录场景的操作步骤

当 `snapshot` 或 `eval` 检测到目标内容因未登录而无法获取时：

1. 向用户明确说明需要登录的网站和原因，并附带风险提示：即使通过 CDP 复用已有浏览器，自动化操作仍可能被网站识别，建议用户评估账号风险后再决定是否登录。
2. 暂停自动化操作，等待用户在浏览器中完成登录。
3. 用户确认后，刷新页面或重新导航，继续执行。

标准话术：

> "当前页面在未登录状态下无法获取 [具体内容]。请在你的浏览器中登录 [网站名]，完成后告诉我继续。"

用户确认登录完成后，不需要重新 attach，直接继续后续操作即可。

## 3. 浏览器内探索流程

以交付用户结果为核心，同时自然地偏向可沉淀的实现路径。当沉淀友好的方式不会明显增加成本时，优先采纳；当它拖慢任务进度时，先完成再评估。

### 第一步：快速探测

attach 或导航到目标页面后，先用 `eval` 快速摸清局面：

- 目标内容是否已在 DOM 中
- 是否需要交互才能暴露
- 是否有可直接提取的数据接口（JSON-LD、`window.__INITIAL_STATE__` 等）
- 目标元素是否有稳定标识（id、`data-testid`、aria-label 等）

若 `eval` 难以判断，用 `snapshot` 辅助理解页面结构。此步骤只负责"看清局面"，不提取数据。

### 第二步：执行任务

用最高效的方式完成用户请求。选择标准只有一个：在当前页面状态下，哪种方式最快、最稳拿到结果。

| 场景 | 推荐方式 | 理由 |
|------|---------|------|
| 内容在 DOM 中，结构清晰 | `eval` | 直接提取，无需交互 |
| 需要交互，且元素定位明确 | `run-code` | 可处理复杂逻辑和多步骤 |
| 页面复杂、元素难定位，或需快速试错 | `snapshot` + 原生命令 | 直观、快，适合不确定场景 |
| 只需验证某个元素是否存在 | `snapshot` + `click` | 最小代价验证 |

关键原则：**不要为了让代码"以后能复用"而牺牲当前任务的完成速度。** 用户等待的是结果。

### 第三步：沉淀评估（任务完成后）

向用户交付结果后，若判断该路径有复用价值，按当前实现形式评估沉淀成本：

| 当前实现 | 沉淀成本 | 动作 |
|---------|---------|------|
| 纯 `eval` 提取 | 低 | 嵌入 `run-code` 的 `page.evaluate()` 即可 |
| `run-code` + 稳定选择器 | 无 | 直接沉淀 |
| `snapshot` + 临时 ref / 原生命令 | 中 | 重写为 `run-code` + 稳定选择器 |
| 混合路径 | 中 | 收敛到 `run-code`，替换临时 ref |

有复用价值时，按上表执行对应动作，通过 `command create` 沉淀。

## 4. 技术事实

以下技术细节在编写 `eval` 表达式和 `run-code` 脚本时经常涉及。

### DOM 中的隐藏内容

页面中存在大量已加载但未展示的内容——轮播中非当前帧的图片、折叠区块的文字、懒加载占位元素等。它们存在于 DOM 中但对用户不可见。以数据结构（容器、属性、节点关系）为单位思考，可以直接触达这些内容。

### Shadow DOM / iframe 边界

DOM 中存在选择器不可跨越的边界：Shadow DOM 的 `shadowRoot`、iframe 的 `contentDocument` 等。`eval` 递归遍历可一次穿透所有层级，返回带标签的结构化内容；但 CSS 选择器和 Playwright locator 默认无法跨越这些边界，需要特殊处理。

### 滚动与懒加载

`/scroll` 到底部会触发懒加载，使未进入视口的图片完成加载。提取图片 URL 前若未滚动，部分图片可能尚未加载，拿到的是占位符地址。

### URL 完整性与参数保留

站点内交互产生的链接通常携带会话相关的参数（如 token），这些参数是正常访问所必需的。提取 URL 时应保留完整地址，不要裁剪或省略参数。手动构造的 URL 可能缺失隐式必要参数，导致被拦截、返回错误页面或触发反爬。

### 反爬与速率控制

- 短时间内密集打开大量页面（如批量 `tab-new`）可能触发网站的反爬风控
- 平台返回的"内容不存在""页面不见了"等提示不一定反映真实状态，也可能是访问方式的问题（如 URL 缺失必要参数、触发反爬）而非内容本身的问题
- 若任务需要借助用户已登录的会话执行操作，开始前须向用户说明风险：使用已登录状态进行自动化操作仍有可能触发网站风控，建议避免在高价值账号上执行

### 媒体资源提取

判断内容在图片里时，优先用 `eval` 从 DOM 直接拿图片 URL，再定向读取——比全页截图精准得多。拿到媒体资源 URL 后，公开资源可直接下载到本地读取；需要登录态才可获取的资源才需要在浏览器内导航 + screenshot。

### 视频内容获取

通过 `eval` 操控 `<video>` 元素（获取时长、seek 到任意时间点、播放/暂停/全屏），配合 `screenshot` 采帧，可对视频内容进行离散采样分析。

## 5. 页面内导航

### 当前 tab 内操作

使用 `click <target>` 在当前页面内直接点击。适合需要在同一页面内连续操作的场景，如点击展开、翻页、进入详情等。简单直接，串行处理。

### 新 tab 打开

使用 `tab-new <url>` 在新标签页中打开目标链接。适合需要同时访问多个独立页面的场景。操作完成后使用 `tab-close` 关闭该标签页。

### 链接参数

很多网站的链接包含会话相关的参数，提取 URL 时应保留完整地址。手动构造 URL 可能缺失隐式必要参数。

---

## 6. 环境整洁

### 创建与关闭标签页

若无明确要求，不主动操作用户已有标签页。任务应在 `tab-new` 创建的后台标签页中进行，保持对用户环境的最小侵入。任务结束后使用 `tab-close` 关闭自己创建的标签页。

### 会话管理

Playwright CLI 通过 CDP attach 复用用户浏览器，连接成功后会话持续保持。不建议主动断开连接——重新 attach 后可能需要用户在浏览器中重新授权远程调试。
