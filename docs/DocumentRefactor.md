# WebSculpt 文档整合方案

> 本文件记录当前 docs/ 下文档的诊断结论、新理念映射，以及整合后的目标结构与实施计划。
> 供审阅与逐条讨论使用。

---

## 1. 整合动机

当前 docs/ 下文档（不含 README.md / AGENTS.md）共 6 份，存在以下结构性问题：

- **重复**：同一概念在多篇文档中反复出现，维护时易遗漏同步
- **过时**：部分文档记录了已修复的问题、已落地的功能、已废弃的设计
- **不全**：Harness 工程、access 封装约束、explore 多工具配合等新理念尚未系统化落地为文档
- **层次模糊**：设计哲学、架构提案、实现纪要、使用手册混在同一层级，读者难以判断阅读顺序

整合目标：**将 6 份文档整合为 5 份核心文档**，消除重复与过时内容，融入新理念，建立清晰的层次关系。

## 3. 新理念映射

### 3.1 Harness 工程

> 本质上，WebSculpt 是一个 Harness 工程（Harness Engineering）的实践。
>
> 传统软件工程是让开发者用代码表达规则；Harness 工程则是用系统化的工程边界约束和引导 AI 的行为，使 AI 的概率性能力在确定性的边界内运作。工程边界的载体可以是代码、文档、契约或运行环境，根据场景需要选择。

现有文档反复提及"确定性的执行层去组织 AI 的概率性能力"，但未使用 Harness 工程这一术语。整合后应在设计总纲中明确引入此概念，作为贯穿全文的统一框架。

### 3.2 access 层 = 工具封装与行为约束

**现有定位**："代码确保工具就绪，并提供统一的环境状态查询接口。AI 直接操作工具。"

**新理念定位**：access 层不仅确保工具就绪，更要**约束 agent 使用工具的行为**。例如 playwright-cli 功能强大，但 access 层通过 `guide.md` 限定 AI 可使用的命令范围、操作规范和风险提示，使 agent 不直接调用 playwright-cli 的底层 API，而是在受控的边界内操作。

文档约束本身就是 Harness 的有效形式。是否以及何时需要代码层封装，取决于实际约束强度需求，不应将代码封装预设为唯一或最终形态。

**文档体现**：在架构文档中重新定义 access 的职责边界，从"做什么/不做什么"升级为"约束什么/暴露什么/以何种载体约束"。

### 3.3 explore 层 = 多工具配合策略

**现有定位**："策略文档层。告诉 AI 面对不同场景时应该优先用什么工具。"

**新理念定位**：explore 层不仅指导"选哪个工具"，更指导"**如何组合多个工具完成复杂任务**"。例如：先用 fetch 探测页面结构，若遇反爬则切换至浏览器自动化，信息获取完成后用本地工具做数据清洗——这是一个多工具配合的流水线，explore 层应提供此类编排策略。

**文档体现**：在架构文档 explore 章节中增加"多工具配合模式"小节，举例说明常见编排策略。

### 3.4 CLI = Agent 经验固化与复用框架

**现有定位**："命令的发现、执行与管理入口。"

**新理念定位**：CLI 框架是 agent 将过去任务中探索出的经验**固化为可复用代码**，并在后续任务中**约束自身使用这些固化代码**的系统。它不只是命令库，更是 agent 的"外置记忆 + 行为约束层"。

**文档体现**：
- `Design.md` 中更新 WebSculpt 定义，从"CLI 框架"升级为"基于 Harness Engineering 理念的信息获取框架"
- `CLI.md` 前言中加入此定位，帮助用户理解 CLI 的存在意义不仅是"管理命令"

### 5.2 `docs/design/Architecture.md`（架构详解）

**定位**：面向开发者和 Agent 的架构手册。回答"系统如何组织、各层职责边界、如何交互、代码在哪里"。是设计总纲的技术展开。

**合并来源**：
- `DESIGN_PROPOSAL.md`：三层架构框架、目录规划
- `CompileDesign.md`：compile 层已确定的设计决策、运行时契约摘要、代码路径速查

**内容结构（建议）**：

```
1. 架构总览
   - 简述三层 + CLI 的关系
   - 引用 Design.md 的控制流图（不复制，用文字描述或链接）

2. access — 工具封装与约束层
   - 职责：不仅是"确保工具就绪"，更是"封装工具能力、约束使用行为"
   - 做什么：提供受控的工具调用面、环境状态查询、操作参考文档
   - 不做什么：不做路由决策、不编排操作序列（这是 explore 的职责）
   - 约束形式：文档约束（guide.md）是 Harness 的有效形式，代码封装是可选增强
   - 统一状态接口（可选设计方向，当前未实现）
   - 目录结构规范

3. explore — 多工具配合策略层
   - 职责：为 AI 提供综合探索策略、工具选择倾向、启发式规则
   - 新增重点：多工具配合模式（如 fetch → 浏览器自动化 → 数据清洗的流水线）
   - 核心特征：产出是 AI 的决策依据，不是机器可执行代码

7. 目录规划与代码路径速查
   - 保留实用参考表格

8. 待确定事项（可选，若仍有悬而未决项）
```

## 8. 实际讨论中的关键修正（执行记录）

以下为本方案原始计划与实际执行之间的重大偏差，供后续步骤参考：

### 8.1 角色命名与映射修正

- **"静态固化者"更名为"构建者"**：原名称与 WebSculpt 控制流中的"固化"概念冲突，且"静态"二字与 AI 时代特征不符。
- **WebSculpt 的角色映射**：原计划说 WebSculpt 承载"静态固化者"与"认知接口"的底座，实际确认 WebSculpt 的核心行为属于**动态适配者**（运行期间直接编写代码以扩展/修正系统功能）。
- **"构建者"的准确定义**：不是"把探索结果转化为代码"，而是"辅助软件开发者编写代码，将规则转化为可执行程序"。

### 8.2 Harness 工程定义修正

- 从"用代码框架约束和引导 AI"改为"用**系统化的工程边界**约束和引导 AI"。
- 工程边界的载体可以是代码、文档、契约或运行环境，根据场景需要选择。
- **删除"马具"比喻**：设计文档中不再使用此文学化表述。

### 8.3 Access 层理念修正

- 原计划暗示 access 层"从纯文档约束走向代码层封装"，将代码封装预设为最终形态。
- 实际确认：**文档约束（guide.md）本身就是 Harness 的有效形式**，不应被贬低为"尚未代码化的过渡态"。是否以及何时需要代码层封装，取决于实际约束强度需求。

### 8.4 Design.md 结构调整

- 原计划的 7 节结构被压缩为 **5 节**，删除了"当前阶段与已知边界"章节（移至 Architecture.md）。
- **问题域与 Harness 工程互换顺序**：先讲问题（第 3 节），再讲解决方案（第 4 节），阅读更自然。
- 核心命题从"Harness Engineering 在 CLI 方向的一次实践"修正为"基于 Harness Engineering 理念的**信息获取框架**"，避免将 CLI 误认为方向/领域。

## 7. 风险与注意事项

1. **信息丢失风险**：合并多份文档时，某些细节可能被遗漏。本方案已尽量标注每份目标文档的"合并来源"和"删除内容"，执行时应逐段核对。
2. **引用断裂**：删除旧文档后，需检查 `README.md`、`AGENTS.md`、代码注释中是否有引用这些旧文档路径的地方。`DESIGN_PROPOSAL.md` 内部引用的 "Design.md" 因在同一目录下可保持有效；`CompileDesign.md` 引用的 "CLI.md" 需更新为 `docs/reference/CLI.md` 的相对路径。
3. **Agent 发现路径**：`AGENTS.md` 中如有引用旧文档路径，需同步更新。当前 `AGENTS.md` 正文未引用具体文档路径，但未来 Agent 可能会读取 docs/ 下文件。
4. **Git 历史**：移动/删除文件会改变 Git 的 blame 路径。如需保留历史可追溯性，可考虑 `git mv`，但重写内容较多时 blame 意义有限。
