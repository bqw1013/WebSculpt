# WebSculpt 设计文档

## 1. 核心命题

**WebSculpt 是一个基于 Harness Engineering 理念构建的信息获取框架。它以 CLI 为交互形态，供 AI 和人类共同沉淀、复用、修复信息获取命令，用确定性的执行层去组织 AI 在信息收集上的概率性能力。**

## 2. 设计哲学

软件系统是演绎的：规则先行，输入确定则输出确定。它负责边界、约束、状态、执行与审计。

大模型是归纳的：无需预设全部规则，通过推断逼近合理结果。它负责理解、归纳、补全、判断与翻译。

真正成熟的系统需要两者的分工：AI 负责"想"，系统负责"做"；AI 负责"提出"，系统负责"验证、批准和执行"。

将这套分工映射到软件系统中，AI 扮演三种典型角色：

| 角色 | 所处阶段 | 核心作用 |
|------|---------|---------|
| **构建者** | 开发期 | 辅助软件开发者编写代码，将规则转化为可执行程序 |
| **动态适配者** | 运行期 | 在系统运行期间直接编写代码，为系统新增或调整功能 |
| **认知接口** | 交互边界 | 将模糊意图翻译为系统可执行的结构化动作 |

构建者的价值在于帮助开发者将规则转化为确定性的软件资产；动态适配者的价值在于让系统在使用过程中直接编写代码以新增或调整自身功能，无需回到开发期；认知接口的价值在于降低精确表达的成本。

WebSculpt 承载的是动态适配者角色：它在用户使用过程中通过探索、沉淀、复用与修复的持续循环，不断适配信息获取的需求变化。

## 3. 问题域：信息收集

在 AI 时代，人们本质上期望 AI 能够替代自己做决策。但让 AI 直接进行决策时，它通常会自主决定如何收集信息并做出判断，这往往导致过程不可控。

信息收集是决策的地基，且极度依赖用户所处的具体场景与先验知识。因此，**信息收集、数据过滤与最终决策必须在架构上实现解耦**。理想状态下，信息收集的方式应当根据用户的偏好进行定制，从而避免 AI 在黑盒中盲目探索。

当下常见的信息获取主要依赖 WebSearch 和 WebFetch 接口，特定场景下辅以 curl 或专属 API。当这些常规手段失效，面对需要复用登录状态、多步骤复杂交互等需求时，就必须引入浏览器自动化工具。目前的痛点在于：浏览器自动化的专业门槛较高，如果让 AI 每次面对复杂网页都"现想现做"，不仅成功率难以保障，还会消耗大量的 Token，且执行极其不稳定。

信息收集天然是一个需要动态适配的问题域：目标网站结构持续变化，反爬策略不断演进，同一信息在不同平台上的获取路径千差万别。这意味着任何固化的信息获取逻辑都终将失效，而系统必须具备在运行期间直接生成或调整获取逻辑的能力——这正是 Harness 工程最适合介入的场景。

## 4. WebSculpt 的本质：面向信息获取的 Harness 工程

> Harness 工程的本质是：不再要求开发者用代码逐条表达规则，而是用系统化的工程边界约束和引导 AI 的行为，使 AI 的概率性能力在确定性的边界内运作。

传统软件工程要求人类把所有规则写下来。Harness 工程要求人类划定工程边界，让 AI 在其中运行。

WebSculpt 就是这样一套以 CLI 为形态的 Harness：

- **固化**：AI 或人机协同在信息获取任务中跑通一次后，将工作流程沉淀为可复用的扩展命令，注册到本地命令库后即成为一条低成本的确定性快路径。
- **复用**：在后续同类任务中，系统约束 agent 调用已沉淀的命令，而不是从头重新探索。这不是在替代 AI，而是为 AI 的概率性能力提供确定性边界。
- **自愈**：当目标网站结构变化导致命令失效时，异常触发修复流程，AI 再次化身动态适配者重构逻辑。

在这个框架下，CLI 不只是一个命令管理入口，更是 agent 的「外置记忆 + 行为约束层」。

## 5. 控制流：探索 → 固化 → 自愈

这不是一张静态模块图，而是一种持续运转的控制流：

```
陌生场景 ──AI 探索──▶ 固化命令 ──直接调用──▶ 多次复用
    ▲                                      │
    └──────── 异常检测 / AI 修复 ──────────┘
```

**核心策略**：第一次靠概率解决，之后尽量把结果沉淀成确定性快路径。等哪天环境又变了、旧逻辑失效了，再重新唤醒 AI 去修。
