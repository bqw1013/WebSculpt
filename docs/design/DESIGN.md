# WebSculpt 设计文档

## 1. 核心命题

**WebSculpt 试图用确定性的执行层去组织 AI 在信息收集上的概率性能力。**

它不是一个通用 Agent，而是一个供 AI 和人类共同沉淀、复用、修复信息获取命令的 CLI 框架。

## 2. 设计哲学：确定性与概率性

传统软件的底层气质是演绎法：开发者先定义规则，再把规则翻译成程序，机器严格执行。理想状态下，给定输入就得到唯一且稳定的输出。

大模型的路径则是归纳法：它不要求事先把规则全部写出来，而是通过统计归纳去逼近一个合理结果。它不保证每次都绝对正确，但在模糊、变化、非结构化的环境里，往往比硬编码的规则更有韧性。

这两者的关系不是替代，而是分工重划：

- **代码负责边界、约束、状态、执行与审计。**
- **AI 负责理解、归纳、补全、判断与翻译。**

真正成熟的系统，一定是 AI 负责"想"，代码负责"做"；AI 负责"提出"，系统负责"验证、批准和执行"。

## 3. 设计哲学：AI 的三种系统角色

如果把这套分工放进软件系统，AI 大概扮演三种典型角色：

| 角色 | 所处阶段 | 核心作用 |
|------|---------|---------|
| **静态固化者** | 开发期 | 将探索结果翻译成可复用的确定性代码、规则或配置 |
| **动态适配者** | 运行期 | 当外部环境变化导致旧规则失效时，局部重构逻辑使系统恢复运转 |
| **认知接口** | 交互边界 | 将用户的模糊意图翻译为系统可执行的结构化动作 |

静态固化者的价值在于把一次性探索变成可复用资产；动态适配者的价值在于让系统在外部变化中保持生命力；认知接口的价值在于降低人机交互的精确表达成本。

## 4. WebSculpt 如何对应这套哲学

WebSculpt 当前主要承载的是「静态固化者」与「认知接口」的底座，「动态适配者」的触发机制尚在探索中。

- **静态固化**：用户或 AI 跑通一次信息获取流程后，将其沉淀为 `websculpt <domain> <action>` 形式的扩展命令。命令由 `manifest.json`（元数据）和 `command.js`（执行逻辑）组成，注册到本地命令库后即可复用。
- **认知接口**：CLI 本身作为结构化入口，未来可能通过自然语言解析命中已有命令或生成新命令。（**待探索**：自然语言到命令的翻译层尚未设计。）
- **动态适配**：当目标网站结构变化导致命令失效时，由异常检测触发修复流程。（**当前未实现**：异常判定标准与修复触发机制均不确定。）

## 5. 问题域：信息收集

在 AI 时代，人们本质上期望 AI 能够替代自己做决策。但让 AI 直接进行决策时，它通常会自主决定如何收集信息并做出判断，这往往导致过程不可控。

信息收集是决策的地基，且极度依赖用户所处的具体场景与先验知识。因此，**信息收集、数据过滤与最终决策必须在架构上实现解耦**。理想状态下，信息收集的方式应当根据用户的偏好进行定制，从而避免 AI 在黑盒中盲目探索。

当下常见的信息获取主要依赖 WebSearch 和 WebFetch 接口，特定场景下辅以 curl 或专属 API。当这些常规手段失效，面对需要复用登录状态、多步骤复杂交互等需求时，就必须引入浏览器自动化工具。目前的痛点在于：浏览器自动化的专业门槛较高，如果让 AI 每次面对复杂网页都"现想现做"，不仅成功率难以保障，还会消耗大量的 Token，且执行极其不稳定。

面对这一困境，需要引入一种**全新的信息收集软件设计理念**——将"动态适配者"与"静态固化者"角色深度结合。在信息获取任务上，让 AI 经历一个"从动态探索到静态固化，再到动态自愈"的演进周期：

1. 面对一个具体的复杂场景，AI 首先通过自主探索或在人类协同下，跑通一次有代表性的获取流程；
2. 随后，AI 将这条概率性的探索路径**静态固化**为一段确定性的代码逻辑，下次遇到同类需求即可直接低成本调用；
3. 当网站变动导致工具失效时，AI 再次化身**动态适配者**重构代码，完成系统局部的自愈。

## 6. Harness 三层能力（框架）

为上述"探索 → 固化 → 自愈"的闭环提供支撑，底层系统需要提供三层能力：

1. **多维探索能力**：让 AI 或人机协同能跑通首次信息获取流程。这要求系统提供恰当的沙箱与工具链协同机制。
2. **经验固化能力**：将跑通的流程转化为可复用的扩展命令，并注册到命令库。这要求系统能够指导 AI 按照规范生成代码，确保固化后的功能不仅对 AI 易于发现，对人类也保持高可读性。
3. **闭环自愈能力**：感知命令失效并触发修复。这要求系统提供完善的监控与自愈组织机制，帮助 AI 精准定位异常，并高效完成逻辑修复。

这三层能力的具体接口、模块划分和数据格式目前均未定型，仅作为框架性目标存在。

## 7. 控制流：探索 → 固化 → 自愈

这三层能力不是静态模块，而是一种持续运转的控制流：

```
陌生场景 ──AI 探索──▶ 固化命令 ──直接调用──▶ 多次复用
    ▲                                      │
    └──────── 异常检测 / AI 修复 ──────────┘
```

**核心策略**：第一次靠概率解决，之后尽量把结果沉淀成确定性快路径。等哪天环境又变了、旧逻辑失效了，再重新唤醒 AI 去修。

这个循环的触发条件、状态持久化方式、命令版本管理策略等细节目前尚未确定。

## 8. 当前阶段与已知边界

### 已实现

- **CLI 骨架**：基于 Commander.js 的 CLI 入口与命令路由已完成。
- **命令扩展机制**：支持用户自定义命令（`~/.websculpt/commands/`）和内置命令（`src/cli/builtin/`）的分层查找。
- **元命令**：
  - `websculpt config init` — 初始化用户目录
  - `websculpt command list` — 列出可用命令
  - `websculpt command show <domain> <action>` — 已注册，占位实现
  - `websculpt command remove <domain> <action>` — 删除用户自定义命令
- **扩展命令示例**：`websculpt example hello` — 供开发者和 AI 参考的模板命令。
- **输出契约**：扩展命令默认以 JSON 输出（便于程序化和 AI 消费），并追加写入 `~/.websculpt/log.jsonl`；元命令默认以人类可读文本输出，可通过全局选项 `--format <human|json>`（`-f` 简写）切换为结构化 JSON。

### 不确定 / 未实现

- **Web 访问能力**：尚未集成任何浏览器自动化或 HTTP 请求工具。库选型未定。
- **命令自动生成**：AI 还不能通过 CLI 直接创建新的扩展命令。入口形态未定（是元命令还是自然语言接口？）。
- **自愈机制**：异常检测标准、修复触发条件、命令版本策略均未设计。
- **参数形态**：当前仅支持 `--key <value>` 形式的 options，positional arguments 的支持方式未定。
- **运行时支持**：`command create` 允许声明 `shell` / `python` 运行时，但 `command-runner` 目前只实现了 `node`。
