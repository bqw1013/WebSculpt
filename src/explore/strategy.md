# WebSculpt 探索策略

> 本文档面向 AI Agent，说明在面对陌生信息获取场景时，如何根据任务特征选择工具、制定探索策略，并为后续固化沉淀出稳定、可复用的命令资产。

---

## 1. 核心哲学：目标导向的探索

探索不是"按预定步骤执行"，而是"带着目标进入，边看边判断，遇到阻碍就解决，发现内容不够就深入"。全程围绕"我要达成什么"做决策。

### 四步决策框架

**① 定义成功标准**
先明确用户要做什么。什么算完成了？需要获取什么信息、执行什么操作、达到什么结果？这是后续所有判断的锚点。

**② 选择起点**
根据任务性质、平台特征、达成条件，选一个最可能直达的工具作为第一步。一次成功最好；不成功则在③中调整。

**③ 过程校验**
每一步的结果都是证据，不只是成功或失败的二元信号。用结果对照①的成功标准，更新你对目标的判断：
- 路径在推进吗？
- 结果的整体面貌（质量、相关度、量级）是否指向目标可达？
- 发现方向错了立即调整，不在同一个方式上反复重试。

搜索没命中不等于"还没找对方法"，也可能是"目标不存在"；API 报错、页面缺少预期元素、重试无改善，都是在告诉你该重新评估方向。

遇到弹窗、登录墙等障碍，判断它是否真的挡住了目标：挡住了就处理，没挡住就绕过——内容可能已在页面 DOM 中，交互只是展示手段。

**④ 完成判断**
对照定义的成功标准，确认任务完成后才停止，但也不要过度操作，不为了"完整"而浪费代价。

## 2. 工具选择策略

### 决策矩阵

| 场景 | 优先工具 | 说明 |
|------|---------|------|
| 需要发现信息来源、获取搜索摘要 | WebSearch | 信息定位入口 |
| URL 已知，需要从页面定向提取特定可读内容 | WebFetch / curl | 静态层直接获取 |
| URL 已知，需要原始 HTML 源码（meta、JSON-LD 等结构化字段） | curl | 获取未经处理的原始响应 |
| 反爬严格的平台（小红书、微信公众号等），静态方式已知无效 | 浏览器自动化 | 跳过静态层，直接访问主站 |
| 需要登录态、多步骤交互、或像人一样自由导航探索 | 浏览器自动化 | 复用用户浏览器会话 |

### 启发式规则

**先轻后重**
优先尝试 WebSearch、WebFetch 或 curl。只有在确认静态方式不可达（遇反爬、需交互、需登录态）时，才启用浏览器自动化。

**一手信息优先**
搜索引擎和聚合平台是信息**定位**工具，不是**证明**工具。找到来源后，直接访问原始页面读取原文。多个二手来源引用同一个错误会造成循环印证假象。

## 3. 浏览器内操作策略

> 当你根据前面章节的工具选择策略判断需要启用浏览器自动化时，请阅读 `src/access/playwright-cli/guide.md` 的连接步骤、操作模式与适用场景、以及技术事实。
>
> 注意：一个任务不必绑定单一工具，可以根据不同阶段的需求灵活组合。

进入浏览器自动化层后，有两种操作思路。根据对目标平台的了解灵活选择。

### 用户行为模拟

通过浏览器的交互 API（点击、填写、按键等）模拟真实用户的操作序列。

- 有自动等待和状态检查，对网站来说更像是正常用户行为
- 反爬触发概率较低
- 步骤多、速度相对慢

### 程序化探测

通过直接操作 DOM（如 `page.evaluate`）或构造 HTTP 请求来跳过中间交互过程。

- 速度快、结果精确
- 可能缺失某些隐式必要参数（如 URL 中的 token、表单中的 hidden field），导致被拦截或返回错误页面

### 决策优先级

1. **任务成功优先**
   任何情况下，功能正确是第一目标。

2. **优先产出可复用的确定性代码**
   使用稳定选择器配合用户行为模拟；或在程序化探测中发现更轻量的确定性路径（如直接请求的 API 接口）时，优先采纳。

3. **用户行为模拟作为程序化方式的有效探测**
   通过一次真实交互观察站点的实际行为（URL 模式、必需参数、页面跳转逻辑），为后续程序化操作提供依据；同时当程序化方式受阻时，用户行为模拟是可靠的兜底。

4. **禁止在可固化代码中使用临时标识**
   依赖临时 snapshot ref 的交互对固化毫无价值——这些标识仅在当前会话有效，下次执行会重新分配。如果探索中不得不使用临时标识完成某步操作，应意识到这段逻辑不可固化，需在固化时用稳定选择器重新实现。

## 4. 探索与固化的衔接：选择稳定路径

探索的终点不是"跑通了"，而是"找到了一条在目标站点当前环境下可持续运行的路"。

### 稳定性优先级（从高到低）

| 优先级 | 获取方式 | 为什么稳定 |
|--------|---------|-----------|
| 1 | 直接数据接口（程序化探测中发现的 API endpoint） | 绕过前端渲染层，不受 UI 改版影响 |
| 2 | 基于语义选择器的程序化探测（遍历特定 `data-testid`、`aria-label` 等属性） | 前端结构微调通常保留语义标识 |
| 3 | 基于稳定选择器的用户行为模拟（使用智能定位器，如 `getByRole` / `getByText`） | 定位器 API 有一定容错能力 |
| 4 | 基于 DOM 位置的硬编码选择器（`:nth-child`、`div > div > span`） | 前端微调极易失效，仅作兜底 |

### 反爬友好度的考量

- **程序化探测速度快但痕迹明显**：高频 DOM 操作或快速翻页容易触发风控。探索阶段就要测试"什么节奏下站点会拒绝响应"，把这个节奏作为固化命令的默认参数。

- **用户行为模拟是反爬友好的基底**：交互操作有自动等待和真实事件流。但固化后如果批量调用（如连续翻 50 页），仍然可能被识别。探索时要验证单链路的可行性，也要估计批量调用的风险。

- **有些平台本身就是浏览器层才有内容**（如小红书、微信公众号）：静态方式完全无效，只能走浏览器自动化。探索时就要确认"必须用浏览器"这个事实，固化命令直接声明 `playwright-cli` runtime，不浪费代码在 fallback 上。

### 不可固化的操作特征

即使当前能跑通，以下特征也意味着固化后很快会失效，探索阶段应主动规避：

- 依赖临时 snapshot ref —— 下次执行标识完全不同
- 依赖"先点 A 再等 2 秒再点 B"的时序硬编码 —— 网络波动即失效
- 依赖未验证的 class name（如 `.css-1a2b3c`）—— 样式重构即失效

### 探索结束的判断标准

找到了一条满足以下全部条件的路径：

1. 使用稳定选择器或确定性接口
2. 在目标站点当前反爬策略下可重复执行
3. 操作序列不依赖临时状态或硬编码时序

满足以上条件后，探索阶段结束，进入固化阶段——将这条概率性路径转化为确定性的 `manifest.json` + `command.js`。
